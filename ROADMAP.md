# خارطة طريق تكوين (Takween Roadmap)

## الملخص
هذه الخارطة تنفيذية (Execution Roadmap) ومبنية على واقع أن متطلبات Baa الأساسية أصبحت متاحة.

- **آخر تحديث:** 2026-03-02
- **مرجع Baa:** `v0.4.4.0`
- **نطاق MVP الأول:** Windows-first + Parser صارم + `تهيئة/init`.

## المتطلبات المسبقة من Baa (محسومة)
- [x] الذاكرة الديناميكية (`حجز_ذاكرة`) — متاحة منذ `v0.3.11.0`
- [x] إدخال/إخراج الملفات (`فتح_ملف`, `اقرأ_سطر`, `اكتب_سطر`) — متاحة منذ `v0.3.12.0`
- [x] معاملات سطر الأوامر (`صحيح الرئيسية(صحيح عدد، نص[] معاملات)`) — متاحة منذ `v0.3.12.5`
- [x] استدعاء النظام (`نفذ_أمر`) — متاح منذ `v0.4.1.0`

## Sprint: 4 أسابيع (MVP)

### Week 1 — `تهيئة` + Parser Core
**الهدف:** تأسيس العقد التشغيلي والتهيئة والتحقق الصارم من ملف المشروع.

- [x] تثبيت عقد CLI (Arabic + English aliases)
- [x] تهيئة هيكل كود MVP (`الرئيسية`, `محلل_النصوص`, `منفذ_الأوامر`)
- [x] دعم `تهيئة/init` لإنشاء قالب مشروع
- [x] Parser صارم: مفاتيح غير معروفة = خطأ فوري
- [x] التحقق من الحقول الإلزامية

**معيار القبول:**
- تشغيل `takween تهيئة` ينشئ `مشروع.تكوين` + `المصدر/الرئيسية.baa`.
- `تحليل_ملف_تكوين` يفشل عند أي مفتاح غير مدعوم أو قيمة فارغة في حقل إلزامي.

### Week 2 — `بناء/build` الأساسي
**الهدف:** ترجمة مشروع صالح باستخدام ملف `مشروع.تكوين`.

- [ ] تحويل المخرجات من parser إلى Build Plan
- [ ] بناء أمر التصريف من `الملفات` + `التحسين` + `المخرج`
- [ ] تنفيذ الأمر وإظهار نتيجة واضحة للمستخدم
- [ ] رسائل خطأ عملية عند فشل التصريف

**معيار القبول:**
- مشروع مثال متعدد الملفات (باستخدام `ملفات`) ينجح في البناء.

### Week 3 — `تشغيل/run` + `تنظيف/clean`
**الهدف:** إكمال دورة التطوير اليومية.

- [ ] `تشغيل/run`: بناء ثم تشغيل الناتج
- [ ] `تنظيف/clean`: حذف مخرجات البناء فقط
- [ ] توحيد أكواد الإرجاع ورسائل الفشل

**معيار القبول:**
- `run` يعكس كود خروج التطبيق المبني.
- `clean` لا يلمس ملفات المصدر أو الإعداد.

### Week 4 — Hardening + توثيق متزامن
**الهدف:** تثبيت MVP بجودة تشغيلية وتوثيق مطابق للسلوك الفعلي.

- [ ] تحسين التشخيص (رسائل أدق + سياق أفضل)
- [ ] توحيد سلوك المسارات على ويندوز
- [ ] إغلاق فجوات التوثيق/التنفيذ
- [ ] قائمة فحص إصدار MVP

**معيار القبول:**
- مرور سيناريوهات الاختبار اليدوي الأساسية وتطابق README/FORMAT/CLI مع التطبيق.

## قرارات تصميم مثبتة
- **لغة الأوامر:** العربية أولاً + aliases إنجليزية.
- **النطاق المنصّي:** Windows-first في MVP.
- **Strictness:** parser صارم (Fail-Fast).
- **اكتشاف الملفات في MVP:** بدون directory walk؛ الاعتماد على `ملفات:` صراحة.

## المخاطر الحالية والتخفيف
- **لا يوجد API أصلي للمشي على المجلدات في Baa:**
  - التخفيف: `ملفات` إلزامي في MVP.
- **تفاوت shell behavior في أوامر النظام:**
  - التخفيف: عزل التنفيذ في `منفذ_الأوامر` وتوحيد الصياغة.
- **انحراف التوثيق عن الواقع:**
  - التخفيف: قفل التوثيق في Week 4 مع فحص قبول موحد.

## ما بعد MVP
- دعم Linux كهدف متكافئ.
- بناء ذكي (Incremental + DAG).
- مدير حزم.
- Dogfooding لبناء أجزاء من Baa عبر Takween.
